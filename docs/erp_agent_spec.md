# ERP 分析智能体需求规格说明书

> 本文档通过访谈方式逐步构建，每 5 轮问答更新一次。

---

## 一、项目背景

基于 LangChain 1.2.0 + 本体论（OWL/SWRL），结合 Oracle EBS 表结构，实现企业级 ERP 分析智能体系统。系统采用五层架构：本体层、知识存储层、编排控制层、专业执行层、工具接入层。

### 第二轮（Q6-Q10）：架构策略与技术选型

---

**Q6：Orchestrator 任务分解粒度**

> 问：P2P MVP 阶段，Orchestrator 对任务的分解粒度策略？

**答：粗粒度 + 预留扩展接口（推荐方案 A）**

- MVP 阶段：Orchestrator 负责意图识别和路由，P2P Agent 内部自行管理子任务拆解（串行）
- 扩展点：Orchestrator 与 P2P Agent 之间定义清晰的**任务接口协议**，未来可将 P2P Agent 内部替换为并行调度器，接口不变

```
Orchestrator
  └── route(intent) → AgentTask(type, params, context)
        └── P2PAgent.run(task)        ← MVP：内部串行
              ↑
              将来可替换为 P2POrchestrator 并行调度
```

**理由：** 细粒度并行调度（DAG）需解决并发合并、局部失败重试等复杂问题，MVP 阶段引入此复杂度得不偿失。

---

**Q7：本体上下文注入形式**

> 问：P2P Agent 的本体 RAG 上下文以何种形式注入 LLM？

**答：混合模式**

| 知识类型 | 注入形式 | 示例 |
|---------|---------|------|
| 业务规则、KPI 阈值 | 结构化 JSON | `{"rule": "three_way_match", "tolerance": "5%"}` |
| 业务背景、流程描述 | 自然语言段落 | "P2P 流程包含采购申请、订单审批、收货确认……" |

**Q8：写操作范围**

> 问：MVP 阶段是否需要实现写操作（付款冻结、创建工单、发送通知等）？

**答：MVP 纯分析只读**
- MVP 阶段：仅读取数据、分析异常、生成报告，不执行任何 ERP 写操作
- 扩展预留：写操作接口（`WriteOperation` 抽象层 + Human-in-the-loop 审批流）在架构中预留，不在 MVP 实现

**Q17：三路匹配容差规则**

> 问：三路匹配偏差容差（默认 5%）是固定值还是可配置？

**答：支持配置化**，按以下维度分别配置：

| 配置维度 | 示例 |
|---------|------|
| 全局默认容差 | 5% |
| 按供应商配置 | 战略供应商可放宽至 8% |
| 按物料类别配置 | 大宗原材料可放宽至 10% |
| 按金额区间配置 | 金额 > 100 万时收紧至 2% |

配置存储于 `config.yaml`，本体 SWRL 规则读取配置值执行校验。

---

**Q18：供应商绩效 KPI 指标**

> 问：供应商绩效评估覆盖哪些 KPI？

**答：全部覆盖**

| KPI | 定义 | 数据来源 |
|-----|------|---------|
| 交货准时率（OTIF） | 按时按量交货订单数 / 总订单数 | `RCV_TRANSACTIONS` vs `PO_LINES_ALL` |
| 发票准确率 | 无差异发票数 / 总发票数 | `AP_INVOICES_ALL` vs `PO_HEADERS_ALL` |
| 质量合格率 | 验收通过数量 / 总收货数量 | `RCV_TRANSACTIONS`（含拒收记录）|
| 价格执行率 | 实际价格与合同价格符合的行数 / 总行数 | `PO_LINES_ALL` vs 合同价格表 |

**Q19：异常记录输出结构**

> 问：每条异常记录包含哪些字段？

**答：由 AI 设计，满足业务场景**

  "anomaly_type": "THREE_WAY_MATCH_VARIANCE",
  "severity": "HIGH",

    "gr_number": "GR-2024-05521",
    "invoice_number": "INV-2024-33210"
  },
  "rule_id": "RULE_P2P_THREE_WAY_MATCH_001",
  "details": {
    "field": "amount",
    "po_value": 100000.00,
    "gr_value": 100000.00,
    "invoice_value": 106500.00,
    "variance_pct": 6.5,
    "tolerance_pct": 5.0
  },
  "description": "发票金额超出采购订单金额 6.5%，超过配置容差 5%",

### 第五轮（Q21-Q25）：知识图谱、记忆与会话管理

---

**Q21：Neo4j 知识图谱数据来源**

> 问：MVP 阶段 Neo4j 中的数据从哪里来？

**答：本体 Schema + 业务实体数据都入图**

| 层次 | 内容 | 来源 |
|------|------|------|
| Schema 层 | 本体类定义、关系类型、属性约束 | Owlready2 解析 `.owl` 文件后同步至 Neo4j |
| 实体层 | 供应商、PO、GR、Invoice 等业务节点及关系 | 模拟数据生成器写入后同步至 Neo4j |

---

**Q22：多轮会话支持**

> 问：MVP 阶段是单轮问答还是多轮会话？

**答：支持多轮会话，短期记忆和长期记忆都要支持**

| 记忆类型 | 作用 |
|---------|------|
| 短期记忆 | 当前会话上下文，支持追问和上下文引用 |
| 长期记忆 | 跨会话保留用户偏好、历史分析结论、常用查询模式 |

---

**Q23：长期记忆存储**

> 问：长期记忆持久化存储方案？

**答：PostgreSQL**

| 维度 | 说明 |
|------|------|
| 存储 | PostgreSQL |
| 内容 | 用户偏好、历史分析结论摘要、常用查询模式、重要异常记录 |
| 与短期记忆关系 | 短期记忆压缩后的摘要定期持久化至 PostgreSQL |

---

**Q24：短期记忆上下文超长处理**

> 问：多轮对话上下文超长时如何处理？

**答：摘要压缩 + 语义检索混合模式**

```
当前上下文构建流程：
  1. 近 N 轮对话（滑动窗口，原文保留）
       +
  2. 早期对话摘要（LLM 定期压缩，存入 PostgreSQL）
       +
  3. 语义相关历史轮次（Chroma 向量检索，按当前问题相关性召回）
  ─────────────────────────────────────────
  → 注入 Agent system prompt 的最终上下文
```

---

**Q25：会话隔离策略**

> 问：多用户并发时会话数据如何隔离？

**答：按 `user_id` 隔离**

| 记忆类型 | 隔离粒度 | 说明 |
|---------|---------|------|
| 短期记忆 | `session_id` | 每次会话独立，会话结束后压缩归档 |
| 长期记忆 | `user_id` | 同一用户跨会话共享，不同用户完全隔离 |
| 知识图谱 | 共享（只读） | 业务知识图谱全局共享，不按用户隔离 |

**Q27：错误处理与可观测性**

> 问：工具失败、LLM 异常时如何处理？是否需要可观测性接入？

**答：自动重试 + 结构化错误响应 + 结构化日志**

错误处理策略：

| 机制 | 说明 |
|------|------|
| 自动重试 | 工具调用失败最多重试 3 次，指数退避间隔 |
| 结构化错误响应 | 任何错误均返回标准 JSON（含错误码、描述、已完成的部分结果） |
| 部分结果返回 | 多子任务场景下，已完成子任务的结果仍返回，不因单个失败丢弃全部 |

标准错误响应结构：
```json
{
  "status": "partial_success",
  "error": {
    "code": "TOOL_QUERY_FAILED",
    "message": "查询供应商数据超时",
    "retry_count": 3
  },
  "partial_results": { ... },
  "completed_tasks": ["three_way_match", "price_variance"],
  "failed_tasks": ["supplier_performance"]
}
```

可观测性：
| 组件 | 选型 | 说明 |
|------|------|------|
| 结构化日志 | `structlog` | JSON 格式日志，含 trace_id、user_id、session_id、耗时 |
| 链路追踪 | `structlog` 手动埋点 | Agent 每个步骤记录输入输出，MVP 不引入独立追踪服务 |

### 第七轮（Q31-Q35）：规则粒度、语言与业务约定

---

**Q31：SWRL 规则粒度**

> 问：采购价格差异和供应商绩效的规则用 SWRL 还是 Python 实现？

**答：混合模式**

| 规则类型 | 实现方式 | 示例 |
|---------|---------|------|
| 核心合规规则 | SWRL（本体定义） | 三路匹配容差校验、付款条款合规判断 |
| KPI 计算与绩效评分 | Python 代码 | OTIF 计算、价格差异率、发票准确率 |

**原则：** 有明确业务合规要求、需要可审计的规则用 SWRL；数值计算、统计汇总用 Python。

**Q32：报告语言**

> 问：分析报告和异常描述默认语言？

**答：默认中文**

- Agent 生成的 Markdown 报告、异常描述、建议动作均使用中文
- 技术字段（字段名、规则 ID、枚举值）保留英文

**Q33：API 认证**

> 问：REST API 是否需要认证机制？

**答：MVP 不做认证**

- MVP 阶段内网使用，无需认证
- 扩展预留：接口设计预留 `Authorization` Header 位置，后续可叠加 JWT 认证中间件

**Q34：默认分析时间范围**

> 问：用户未指定时间范围时，默认分析多长时间的数据？

| 配置项 | 默认值 | 说明 |
|-------|-------|------|
| `analysis.default_time_range_days` | 30 | 用户未指定时间时使用 |
| `analysis.max_time_range_days` | 365 | 单次查询最大时间跨度 |

**Q36：付款合规性分析规则**

> 问：付款合规性检测哪些异常类型？

**答：逾期付款 + 提前付款 + 折扣使用合规**

| 异常类型 | 检测逻辑 | 严重等级 |
|---------|---------|---------|
| 逾期付款 | 实际付款日 > 付款条款到期日 | HIGH |
| 提前付款 | 实际付款日 < 付款条款到期日 - 配置天数阈值 | MEDIUM |
| 折扣未使用 | 在折扣有效期内未申请折扣付款 | LOW |
| 折扣滥用 | 超过折扣有效期仍按折扣金额付款 | HIGH |

## 四、MVP 范围边界

### 包含（IN SCOPE）
- 采购 P2P 模块全部四个分析场景
- Oracle EBS 标准表结构本体建模（12 张核心表）
- 基于 Owlready2 的 SWRL 合规规则执行
- Neo4j 知识图谱（Schema + 业务实体数据）
- Chroma RAG 双路索引（本体知识语义检索）
- GLM-4 驱动的 P2P Agent（配置化模型）
- 多轮会话（短期记忆摘要压缩 + 长期记忆 PostgreSQL）
- FastAPI REST API（`POST /api/v1/analyze`）
- 模拟数据生成器（低异常率，覆盖全部异常场景）
- 结构化错误响应 + 自动重试
- structlog 结构化日志
- pytest 单元测试 + 集成测试（覆盖率 ≥ 85%）
- Docker 应用镜像

### 不包含（OUT OF SCOPE，预留扩展）
- O2C、财务、库存、生产 MRP 模块
- Orchestrator 细粒度并行 DAG 调度
- ERP 写操作（付款冻结、工单创建、通知发送）
- Multi-Agent 并行协作
- API 认证（JWT / API Key）
- 流式响应（SSE）
- 生产级追踪（OpenTelemetry / LangSmith）
- 向量数据库迁移（Qdrant）
- 报表可视化（图表生成）
