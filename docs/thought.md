## ERP Agent × 本体论 × 大模型 — 整体架构说明

---

## 架构总体思路

整个系统分为五层，从下到上依次是：领域知识层、知识存储层、编排控制层、专业执行层、工具接入层。用一句话概括各层的职责：**本体层定义"什么是对的"，知识层存储"知道什么"，编排层决定"做什么"，执行层完成"怎么做"，工具层负责"拿到数据"**。

---

## 第一层：ERP 本体层（领域知识的根基）

这一层是整个系统的知识地基，用 OWL（网络本体语言）对 ERP 业务领域进行精确建模。它解决的核心问题是：把 ERP 领域专家脑子里的隐性知识变成机器可处理的显式结构。

本体层包含四类知识：

第一类是**业务实体类**，定义 ERP 中所有的核心概念及其层次关系。例如"销售订单"是"订单"的子类，"发票"是"财务凭证"的子类。子类自动继承父类的所有规则和约束，同时可以有自己的专属属性。这种继承体系让 Agent 在分析时能自动获得正确的业务语义，而不是把"发票"和"采购订单"当成两个毫无关联的词汇。

第二类是**关系与属性**，定义实体之间如何相互关联。比如"销售订单由客户下单"、"发票引用采购订单"、"采购订单与供应商关联"。这些关系定义了数据之间的语义连接，Agent 在分析时能沿着这些关系路径追溯问题根因，而不需要人工告诉它数据怎么关联。

第三类是**业务规则**，用 SWRL 规则语言把隐性的业务约束显式化。比如"三路匹配规则"——发票金额与采购订单和收货单的偏差超过 5% 时标记为异常；"信用额度规则"——客户未清余额超过信用额度 90% 时触发预警；"MRP 补货规则"——当库存低于安全库存加上日均需求乘以提前期时触发紧急采购。这些规则一旦在本体中定义，就成为所有 Agent 的统一行为准则。

第四类是**KPI 指标体系**，在本体中精确定义每个财务和运营指标的计算公式、数据来源、行业基准值和异常阈值。比如 DSO（应收账款周转天数）的公式是应收余额除以收入再乘以 365，行业良好基准是低于 30 天，超过 60 天触发预警。这些定义让 Agent 计算指标时有据可查，评判结果时有标准可依。

## 第二层：知识图谱与向量索引层（知识的存储与检索）

本体定义的是结构，这一层负责把结构实例化为可检索的知识。包含四个子系统：

**Neo4j 图数据库**存储所有 ERP 业务实体及其关系，形成知识图谱。每个采购订单、客户、发票、物料都是图中的节点，它们之间的业务关系是图中的边。当 Agent 需要分析"哪些客户的逾期发票来自哪些销售订单"，图数据库能在毫秒级内沿关系路径找到答案，而传统 SQL 的多表 JOIN 在复杂多跳查询时会急剧变慢。

**RAG 向量索引**把本体中的类定义、KPI 定义、业务规则都向量化后存入向量数据库，支持语义检索。当用户问"分析应收账款逾期"，系统能自动检索到相关的 DSO 计算规则、账龄分类标准、信用风险评估规则，注入 Agent 上下文。这里采用双路索引——稠密向量捕捉语义相似性，BM25 稀疏索引捕捉关键词精确匹配，两者互补，确保"DSO"、"三路匹配"这类专业术语也能被准确召回。

**规则引擎（Drools）**负责实时执行本体中定义的业务规则。Agent 把业务数据提交给规则引擎，规则引擎返回哪些规则被触发、触发了哪些异常。这把规则执行从 LLM 的概率性判断中剥离出来，变成确定性的逻辑计算，保证合规检查结果的一致性和可审计性。

**ERP 实时数据连接**通过标准接口（SAP BAPI、Oracle SOA 等）获取 ERP 系统中的真实业务数据，并维护一层缓存（ODS 层），避免 Agent 的高频查询直接冲击 ERP 核心生产系统。

## 第三层：Orchestrator 编排层（分析任务的大脑）

这一层是整个 Agent 系统的控制中枢，负责接收用户的分析请求，规划分析任务，调度各专业 Agent 执行，最后聚合结果。

**意图解析与场景路由**是 Orchestrator 接收请求后的第一步。它先从本体 RAG 中检索相关业务背景，然后解析用户意图，判断属于哪种分析场景（P2P 采购分析、O2C 销售分析、库存分析、财务合规分析还是跨模块综合分析），以及分析的深度（概览、问题定位、根因分析还是趋势预测）。这一步的准确性直接决定后续整个分析流程的方向。

**分析任务规划**把一个分析请求分解为若干可并行执行的子任务，并构建子任务之间的依赖关系图（DAG）。例如分析应收账款逾期问题时，系统会同时启动：获取未清发票、获取客户历史付款记录、获取客户信用档案、执行业务规则校验这四个子任务。这四个任务没有相互依赖，可以并行执行。之后的账龄分析和风险评分需要等前面的数据任务完成才能开始。通过这种并行调度，分析总时间可以从串行的四五十秒降低到十几秒。

**本体上下文注入**在每个子任务分发给 Sub-Agent 之前，从本体 RAG 中检索该任务相关的业务知识，作为结构化上下文注入 Sub-Agent 的 system prompt。每个 Sub-Agent 只获得与自己任务直接相关的本体知识，而不是完整的 ERP 本体——这既避免了上下文污染，也节省了 token 消耗。

**结果聚合**在所有 Sub-Agent 完成后，收集所有分析结果，做一致性校验（确保不同 Agent 的发现没有互相矛盾），然后生成结构化的分析报告，包括执行摘要、具体发现（每条发现都标注了依据的本体规则 ID，保证可追溯）、KPI 仪表盘和行动建议。

## 第四层：专业 Sub-Agent 层（领域专家的模拟）

这一层由多个专业化的 Sub-Agent 组成，每个 Agent 深耕一个 ERP 模块，有专门定制的 system prompt 和工具集。

**采购 Agent（P2P）**专注于采购到付款全流程。它负责三路匹配异常检测（采购订单、收货单、发票三者的数量和金额比对）、采购价格差异分析（实际采购价格与标准价格的偏差）、供应商绩效评估、付款合规性检查。它的工具集包括查询采购订单、查询收货记录、执行三路匹配校验、计算价格差异等。

**销售 Agent（O2C）**专注于订单到收款全流程。它负责应收账款账龄分析、DSO 趋势分析、客户信用风险评估、OTIF（准时完整交货率）分析、开票准确性检查。账龄分层标准（0-30 天正常、31-60 天关注、61-90 天预警、90 天以上高风险）来自本体 KPI 定义，不是 LLM 自己判断的。

**财务 Agent**专注于财务合规与报表分析。它负责凭证借贷平衡验证、成本中心费用合规检查、增值税进销项匹配、期末结账完整性验证。所有计算结果由专用财务计算引擎产生，不依赖 LLM 做数值运算。

**库存 Agent**专注于库存与供应链分析。它负责库存周转率分析、呆滞库存识别（超过 180 天未移动的物料）、安全库存合理性评估、账实差异分析。

**生产 Agent（MRP）**专注于物料需求计划分析。它负责 MRP 运算结果合理性评估、产能瓶颈识别、BOM 准确性验证、生产效率分析。

**规则校验 Agent**是整个系统的"质检员"，专门对其他 Agent 的分析结论进行合规性校验，确保每一条发现都有明确的本体规则依据，防止 LLM 的主观判断混入分析结果。

## 第五层：工具与数据层（数据的安全入口）

这一层提供所有 Agent 可以调用的工具，是 Agent 与真实数据之间的安全隔离层。

**SQL 查询工具**不接受 Agent 直接传入 SQL 语句，而是接受结构化的查询意图和参数，工具层内部生成参数化 SQL 执行。这从根本上防止了提示词注入攻击导致的 SQL 注入风险。所有查询结果在返回前还要做数据脱敏处理，去除手机号、身份证等敏感信息。

**SPARQL 本体查询工具**允许 Agent 直接查询本体和知识图谱，获取业务规则、实体定义、KPI 计算公式等结构化知识。同样采用模板化查询，不接受自由 SPARQL 语句。

**ERP API 调用工具**通过官方接口与 ERP 系统交互。关键设计是读写分离：读操作直接执行，写操作（创建单据、过账、发送通知）必须经过 Human-in-the-loop 审批，Agent 只能提出建议，不能自动执行有真实业务后果的操作。

**财务计算引擎**是一个专用的数值计算服务，所有 KPI 计算（DSO、库存周转率、采购价格差异等）都由它执行，结果注入 Agent 上下文。这条原则非常重要：LLM 的浮点计算不可靠，财务数字容不得误差，必须把数值计算从 LLM 的职责中剥离出去。

**报表可视化工具**在 Agent 完成分析后，根据分析结论生成图表（账龄分布柱状图、KPI 仪表盘、趋势折线图等），供决策者直接查看。

## 数据流的完整路径

用一个具体例子串联整个架构：用户请求"分析最近三个月的应收账款逾期风险"。

请求进入 Orchestrator，意图解析识别为 O2C 分析场景，分析深度为问题定位。本体 RAG 检索到 DSO 定义、账龄分类规则、信用风险评估规则。Orchestrator 生成 DAG：并行启动获取未清发票、获取客户信用档案、获取历史付款记录、执行规则校验四个子任务，分别分配给 O2C Agent、O2C Agent、财务 Agent、规则校验 Agent。

每个 Agent 在启动时收到精确的本体上下文（只包含与自己任务相关的业务知识），然后通过工具层调用 ERP 数据：查询未清发票列表、获取客户信用状态、查询付款历史记录。财务计算引擎计算每个客户的 DSO 和逾期金额。规则校验 Agent 把数据提交给规则引擎，获得哪些客户触发了信用预警规则。

四个子任务并行完成后，Orchestrator 聚合结果：生成账龄分布图、识别高风险客户排名、计算总逾期金额、给出催收优先级建议，每条建议都引用了对应的本体规则 ID 作为依据。最终报告返回给用户。

整个过程中，LLM 负责推理和文本生成，本体负责提供准确的业务规则，规则引擎负责确定性的合规验证，财务计算引擎负责精确的数值计算，各司其职，互不越界。
